Overview

This is the central configuration and control hub for all Kive Apps Android applications. It's a static GitHub Pages site that serves as:

  • Public website — Home page, privacy policies, support pages, contact forms
  • JSON API center — Remote configuration endpoints that Android apps fetch to control behavior
  • Management dashboard — Edit JSON files to push announcements, trigger updates, show popups, toggle features — all without releasing a new app version
How it works Your Android apps fetch JSON files from this GitHub Pages site on launch. By editing these JSON files and pushing to GitHub, you instantly control what every user sees — no server, no backend, no cost.

Base URL

https://kiveapps.github.io

API Base

config.json

The primary configuration file. Your Android app should fetch this on every launch to check for updates, maintenance mode, and get current links.

GET /api/v1/apps/lifemaster-ai/config.json

Returns the full app configuration including version info, update requirements, maintenance status, and links.

Key Fields

FieldTypeDescription
version.lateststringLatest version string (e.g. "1.2.0")
version.latest_codeintLatest version code (Android versionCode)
version.min_supportedstringMinimum version that still works
version.min_supported_codeintMinimum supported versionCode
update.mandatoryboolIf true, user MUST update (blocks app)
update.recommendedboolIf true, shows non-blocking update prompt
update.dismiss_for_daysintDays before showing recommended update again
maintenance.enabledboolIf true, show maintenance screen
maintenance.allow_offline_useboolIf true, let users use offline features during maintenance

Android Implementation Logic

// Pseudocode for app startup val config = fetchJson("config.json") // 1. Check maintenance if (config.maintenance.enabled) { showMaintenanceScreen(config.maintenance) if (!config.maintenance.allow_offline_use) return } // 2. Check mandatory update if (appVersionCode < config.version.min_supported_code) { showForceUpdateDialog(config.update) // Non-dismissible return } // 3. Check recommended update if (config.update.recommended && appVersionCode < config.version.latest_code) { showUpdateDialog(config.update) // Dismissible }

How to Trigger a Mandatory Update

  1. Release new version on Play Store
  2. Edit config.json:
    • Set version.latest and version.latest_code to new version
    • Set version.min_supported and version.min_supported_code to the new minimum
    • Set update.mandatory to true
    • Update update.message with reason
  3. Push to GitHub — all users see mandatory update on next app open

notifications.json

Remote notification triggers. The app polls this endpoint and shows local notifications based on the entries. This is NOT Firebase push — it's a pull-based system that works without any server.

GET /api/v1/apps/lifemaster-ai/notifications.json

Returns notifications to show. The app should check this periodically (e.g. on app open or via WorkManager) and schedule local notifications.

Notification Object Fields

FieldTypeDescription
idstringUnique ID — track shown notifications locally
titlestringNotification title
bodystringNotification body text
channelstringAndroid notification channel: updates | announcements | reminders | system
action.typestringopen_app | open_screen | open_url | deep_link | open_play_store
schedule.preferred_timestring?Preferred time to show (HH:mm format)
schedule.repeatstringnone | daily | weekly
display.silentboolShow silently (no sound/vibration)
prioritystringlow | default | high | urgent
ttl_hoursint?Auto-expire after N hours (null = never)
activeboolMaster switch
Important Since this is pull-based (not Firebase), notifications will appear when the app runs — not in real-time. For time-critical messages, prefer announcements or popups instead.

Feature flags and remote configuration values. Toggle features, control behavior, and A/B test without releasing new versions.

GET /api/v1/apps/lifemaster-ai/features.json

Returns all feature flags and their values. App should cache and fallback to defaults if fetch fails.

Adding a New App

When you launch a new Android app, follow these steps to add it to the Kive Apps hub:

  1. Create API directory: api/v1/apps/{new-app-slug}/
  2. Copy all 5 JSON files from an existing app's folder and customize them
  3. Create HTML pages (privacy, support, updates) either per-app or shared
  4. Add app icon to the project (e.g. apps/{slug}/icon.png)
  5. Update home page (index.html) to list the new app in the apps grid
  6. Update this docs page to include the new app's API base URL
  7. In the Android app, point API calls to: https://kiveapps.github.io/api/v1/apps/{slug}/

Force Update Flow

Step-by-step guide for pushing a mandatory update to all users:

  1. Publish new APK/AAB to Google Play Store
  2. Wait for Play Store to process (usually 1-4 hours)
  3. Edit api/v1/apps/{slug}/config.json:
    // Update these fields: "version": { "latest": "2.0.0", "latest_code": 10, "min_supported": "2.0.0", // Set to new version for mandatory "min_supported_code": 10 // All older versions blocked }, "update": { "mandatory": true, "message": "This update includes critical fixes. Please update to continue." }
  4. Commit and push to GitHub
  5. Within minutes, all users on older versions will see a non-dismissible update dialog

Soft (Recommended) Update

For non-critical updates, use a softer approach:

"update": { "mandatory": false, "recommended": true, "message": "A new version is available with improvements!", "dismiss_for_days": 3 // Show again after 3 days if dismissed }

Android Integration Guide

How to integrate the remote config system into your Android app.

1. Define API Constants

object KiveApi { private const val BASE = "https://kiveapps.github.io/api/v1/apps/" fun configUrl(slug: String) = "$BASE$slug/config.json" fun announcementsUrl(slug: String) = "$BASE$slug/announcements.json" fun notificationsUrl(slug: String) = "$BASE$slug/notifications.json" fun popupsUrl(slug: String) = "$BASE$slug/popups.json" fun featuresUrl(slug: String) = "$BASE$slug/features.json" }

2. Fetch on App Startup

// In your Application class or splash screen suspend fun fetchRemoteConfig() { try { val config = httpClient.get(KiveApi.configUrl("lifemaster-ai")) val features = httpClient.get(KiveApi.featuresUrl("lifemaster-ai")) val announcements = httpClient.get(KiveApi.announcementsUrl("lifemaster-ai")) val popups = httpClient.get(KiveApi.popupsUrl("lifemaster-ai")) // Cache locally configStore.save(config) featureStore.save(features) announcementStore.save(announcements) popupStore.save(popups) } catch (e: Exception) { // Use cached values — app works offline } }

3. Processing Priority

// Recommended check order on app startup: 1. Kill switch → features.kill_switch 2. Maintenance → config.maintenance.enabled 3. Mandatory update → config.version.min_supported_code 4. Recommended update → config.update.recommended 5. Popups → popups (filtered by targeting) 6. Announcements → announcements (filtered by targeting) 7. Notifications → notifications (schedule locally)

4. Caching Strategy

  • Cache all responses locally (SharedPreferences or DataStore)
  • Use features.remote_config.cache_duration_minutes to control refresh interval
  • Always fall back to cached data if network fails
  • Default values should be hardcoded in the app as ultimate fallback
GitHub Pages CDN GitHub Pages is served via Fastly CDN with good global performance. JSON files are small (<5KB each) so fetching is fast. Cache headers are ~10 minutes by default.
