# Kive Apps — AI Agent Guide

> **Private document.** Excluded from GitHub Pages output.
> **For AI assistants (Cascade, Cursor, Copilot, Claude Code, etc.) working on Kive Apps codebases.**

You are an AI agent assigned to a Kive Apps project. Read this document **first**, then load the linked references on demand. **Do not skip this document — it dictates how you must operate on this codebase.**

---

## 0. Hard Rules

1. **Never expose internal docs.** Files starting with `_` in `kiveapps.github.io` are private. Do not link to them, copy them into public files, or recreate their content in public-facing markdown.
2. **Never hardcode the master secret.** It lives only in GitHub Secrets. If you see a 128-hex-char string in any file you're asked to work on, treat it as a leak and alert the user.
3. **Never weaken encryption.** Don't suggest "easier" fallback paths, plaintext mirrors for testing, or HTTP basic-auth shortcuts. The architecture is final.
4. **Never edit `dist/` output.** That folder is regenerated by CI on every build. Edits will be lost and could leak plaintext.
5. **Never modify path/key derivation logic** without coordinated APK rebuild — old APKs will stop working.

## 3. Common Tasks — Decision Trees

### Task: "Add a banner to the app"
1. Locate `kiveapps-source/_source/<app-slug>/v<version>/announcements.json`.
2. Append a new entry per `_API_REFERENCE.md` §2 schema.
3. Validate JSON syntactically.
4. Commit and push.
5. Done — CI deploys.

### Task: "Disable a feature for all users"
1. Edit `features.json` → set the `global_flags.<flag>` to `false`, **or** add the flag to `rollouts[]` with `percentage: 0`.
2. Commit and push.

### Task: "Enable premium for a specific user"
1. Get the user's `install_id` and `android_id` from your support workflow.
2. Compute their hash: `HMAC-SHA256(USER_HASH_KEY, install_id || android_id)`.
   *(If you can't compute it locally, ship them a one-time debug build that prints their hash, OR ask the user to send it via the support form.)*
3. Take the first 8 hex characters as their prefix.
4. Add a `user_grants` entry in `premium.json`:
   ```json
   "user_grants": {
     "<8-hex-prefix>": {
       "tier": "premium",
       "granted_at_iso": "2026-04-27T10:00:00Z",
       "expires_at_iso": null,
       "note": "Verified purchase #1234"
     }
   }
   ```
5. Commit and push.

### Task: "Implement the Android SDK"
1. Read `_ANDROID_INTEGRATION.md` end-to-end.
2. Read `_ARCHITECTURE.md` §§4–7 (file format, envelope, manifest).
3. Implement `KapFile`, `EnvelopeParser`, `KiveManifestRepo`, `KiveEndpointRepo`, `KiveUserId`, `KiveRefreshScheduler` per the reference.
4. Run the testing checklist in `_ANDROID_INTEGRATION.md` §12.

### Task: "Add a new app"
1. Read `_BUILD.md` §3.
2. Add entry to `_meta/versions.json`.
3. Copy an existing `v1_0_0/` folder, update slug/package/etc.
4. On the public repo, scaffold marketing pages under `apps/<new-slug>/`.

### Task: "Update the public marketing site"
- All marketing HTML lives in `kiveapps.github.io` (the public repo).
- Use shared CSS at `assets/css/shared.css`.
- Use shared JS at `assets/js/shared.js`.
- Pages MUST include:
  ```html
  <meta name="robots" content="index,follow,noai,noimageai">
  <meta name="googlebot" content="index,follow,noai,noimageai">
  ```
- Pages MUST NOT mention any individual developer's name. Brand is "Kive Apps".

### Task: "Investigate why an app isn't picking up a config change"
1. Verify the CI workflow ran successfully (Actions tab).
2. Verify the auto-commit landed on the public repo.
3. Run privacy verification curl commands from `_BUILD.md` §7.
4. Check the app logs for `AEADBadTagException` (key mismatch).
5. Check the app's cache TTL — most endpoints refresh every 3 hours; critical ones every 15 minutes.
6. Force-restart the app to bypass cache.

## 4. Style & Quality Guardrails

When writing code, docs, or content for Kive Apps projects:

- **No vendor names.** Brand everything as "Kive Apps". Never use "developer's name", "Shubham", or any individual identifier.
- **No personal stories** in support/contact pages. Use plural ("the team", "we") not singular ("I", "the dev").
- **No external analytics** without explicit user approval. Privacy is a brand promise.
- **No plaintext API endpoints.** Anything the app fetches goes through the encrypted pipeline.
- **No public docs links** to `_*.md` files. They're private.
- **Cite line ranges** when discussing code (use the `@filepath:line` convention shared with the user).

## 5. Common Pitfalls

| Pitfall | Avoid by... |
|---------|-------------|
| Running `node build.js` and committing `dist/` to public repo | Always let CI handle deployment; `dist/` is `.gitignore`d in source repo |
| Adding a JSON endpoint without updating `_meta/versions.json` `endpoints[]` | Build skips files not listed; endpoint won't appear in manifest |
| Forgetting to bump `version_code` for a new APK | CI builds for an existing version; old APK signature mismatch on integrity |
| Putting plaintext config URLs in the app | Negates the entire encryption layer |
| Not testing AAD validation | Allows endpoint-swap attacks |
| Logging decrypted payloads to disk in plaintext | User's local disk becomes a leak target — use `EncryptedSharedPreferences` |
| Editing `_config.yml` without verifying private docs still 404 | Privacy regression |

## 6. When to Ask the User

Don't ask for permission for routine edits. Do ask before:
- Rotating `MASTER_SECRET`.
- Force-updating users (`min_supported_code` bump).
- Activating the kill switch.
- Changing the path-derivation algorithm.
- Removing a version from `_meta/versions.json`.

---

## 7. Escalation Path

If you encounter:
- **Cryptographic anomaly** (decryption failures across multiple endpoints) → check master secret consistency between repo secrets and what's hardcoded in current APK.
- **CI failures** → first check secrets are set; second check Node 20+ is being used; third check `_meta/versions.json` is valid JSON.
- **Leaked secret in commit history** → see `_BUILD.md` §8 incident response.
- **Public exposure of private docs** → push no-op commit, wait, verify with curl.

## 9. Verification Mantras

Before you push, ask yourself:

1. Does this expose the master secret? → ABORT
2. Does this leak a developer's personal name? → REWORD
3. Does this add a plaintext API endpoint? → REDIRECT through encrypted pipeline
4. Does this remove a `_*.md` from `_config.yml` exclude? → CONFIRM intentional
5. Does this change cipher parameters? → COORDINATE with APK release
